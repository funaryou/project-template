# ドキュメント・アーキテクチャ設計 [#1]

このドキュメントでは、`<project-name>` プロジェクトにおけるドキュメントの配置思想、構造、および管理ルールを定義します。

## 設計理念

ドキュメントは「生きた資産」であり、コードと同様に整理され、検索可能でなければなりません。本プロジェクトでは、開発フェーズ、機能、技術レイヤーごとにディレクトリを分離し、迷子ファイルの発生を防ぎます。

## ディレクトリ構造

ドキュメントは `doc/` ディレクトリ配下に、以下の役割別に分類します。

| ディレクトリ | 役割 | 主な内容 |
| :--- | :--- | :--- |
| `architecture/` | **システム設計** | 全体の構成図、共通設計理念、技術選定の経緯、システムガイド。 |
| `database/` | **データ設計** | ER図、テーブル定義、マイグレーション手順、Drizzleスキーマ解説。 |
| `features/` | **機能仕様・実装** | 各機能（記事編集、アカウント管理等）の具体的な仕様と、個別の実装記録（`impl_*.md`）。 |
| `implementation/` | **開発プロセス** | 環境構築手順、開発フェーズ別の計画書、全般的な実装ガイドライン。 |
| `verification/` | **検証・レビュー** | レビュー結果（`review_*.md`）、検証手順書、フィードバック対応記録、テスト証跡。 |
| `misc/` | **その他** | 一時的なメモ、作業予定、雑多な記録。 |

## ファイル命名規則

一貫性を保つため、以下の命名パターンを推奨します。

| 種類 | 命名パターン | 例 |
| :--- | :--- | :--- |
| **実装記録** | `impl_{feature_name}.md` | `impl_article_search.md` |
| **レビュー記録** | `review_{phase}_{topic}.md` | `review_phase4_auth.md` |
| **手順書** | `phase{N}_{topic}_steps.md` | `phase5_cleanup_steps.md` |
| **解説ガイド** | `{topic}_guide.md` | `supabase_auth_guide.md` |

## ドキュメント運用ルール

1. **配置の原則**: 
   - 新しいドキュメントを作成する際は、必ず上記のディレクトリ構造に従って配置します。
   - ルートディレクトリ（`doc/` 直下）にファイルを放置しないでください。
2. **バージョン管理**: 
   - 各ドキュメントの H1 見出しの末尾に `[#N]` 形式でバージョンを記載します。
   - 内容を更新した場合は、この数値をインクリメントしてください。
3. **LinkShare 連携**: 
   - `doc/` 配下の md ファイルは `<host>:<port>` の LinkShare アプリで閲覧可能です。
   - 更新時は LinkShare API を通じて記事内容を同期することを推奨します（詳細は `wiki-dev.md` 参照）。

## .codex ディレクトリについて

`.codex/` ディレクトリは、AIアシスタント（特に Codex 思想に基づくエージェント）が参照するための、より構造化された知識や「スキル」を格納する場所です。
- `skills/`: 特定のタスクを実行するための `SKILL.md` 指示書。
- `workflows/`: 非定型的な開発ワークフローの定義。
- `resource/`: AIが文脈を理解するための補足リソース。
