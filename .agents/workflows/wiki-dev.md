# プロジェクト開発ワークフロー

## プロジェクト構成

```
<project-root>/
├── doc/                          ← ドキュメント（ガイド・レビュー・手順書）
├── <app-source>/                 ← アプリケーションのソースコード
│   ├── src/
│   │   ├── proxy.ts              ← ルーティング/保護レイヤー（名称はプロジェクト依存）
│   │   ├── lib/
│   │   └── db/
│   ├── docker-compose.yml
│   └── package.json
└── demo/
```

## ドキュメント規約

### 配置場所
- すべてのドキュメントは `doc/` ディレクトリに置く

### ファイル命名規則

| 種類 | 命名パターン | 例 |
|---|---|---|
| 解説ガイド | `{topic}_guide.md` | `auth_guide.md`, `routing_guide.md` |
| レビュー | `review_phase{N}_{topic}.md` | `review_phase3_auth.md` |
| 手順書 | `phase{N}_{topic}_steps.md` | `phase4_route_guard_steps.md` |

### バージョン番号
- 各ドキュメントの **h1 末尾** にバージョン番号 `[#N]` を付ける
- 内容を更新したらインクリメントする
- 例: `# ルート保護実装ガイド [#2]`

## ドキュメント連携

ユーザーはローカルで動くドキュメント連携アプリで md ファイルを閲覧している。

### ドキュメント更新時の手順
1. `doc/` 配下の md ファイルを編集する
2. h1 のバージョン番号をインクリメントする
3. 連携先 API がある場合は article を更新する

### API による更新方法（テンプレート）
```bash
cd <project-root>/doc
curl -s -X PUT http://<host>:<port>/api/articles/{ARTICLE_ID} \
  -H "Content-Type: application/json" \
  -d "$(jq -n --rawfile content {filename}.md '{content: $content}')"
```

### article ID マッピング
- このテンプレートでは固定 ID は持たない
- プロジェクト側で `doc/article-id-map.md` などを作成して管理する
- 新しいドキュメントを登録する場合は POST `/api/articles` を使う

## コーディング規約

### 命名
- アプリ層は可読性重視（例: キャメルケース）
- 永続層は対象 DB/ORM の規約に合わせる（例: スネークケース）

### セッション/トークン受け渡し
- 方式（クエリ、Cookie、Header）はプロジェクト要件で統一する

### 回答言語
- 日本語で回答する

## 例外

### ドキュメント追記を行わないケース
- ユーザーが「チャットでいい」と明示した場合は、ドキュメント更新や連携 API 実行を行わない
