# ガイド系ドキュメント フォーマットルール

> **目的**: 実装ドキュメントを「答えを教える」から「考えながら学べる教科書」に変換するためのルール集
> **対象**: このリポジトリでガイド系ドキュメントを書くすべての開発者

---

## 1. 基本原則

### 1.1 ガイドと答案コードの分離

1つの機能に対して、**2つのファイル** を用意する：

| ファイル | 命名ルール | 内容 | 例 |
|---------|-----------|------|----|
| **ガイド** | `{feature}_guide.md` | 概念解説・関数リファレンス・課題。**完成コードを含めない** | `file-upload-guide.md` |
| **答案コード集** | `{feature}_answers.md` | 全コードの完成形。ファイル単位で整理 | `file-upload-answers.md` |

**補足**: 従来の `handbook.md` のような「解説＋コード一体型」を否定するものではない。学習スタイルに応じて選択できるようにするのが目的。

### 1.2 姉妹ドキュメント間のリンク

ガイドと答案コード集は冒頭で相互リンクする：

```markdown
<!-- ガイドの冒頭 -->
> **📝 答案コード集はこちら**: [{feature}_answers.md]({feature}_answers.md)（行き詰まったら参照）

<!-- 答案コード集の冒頭 -->
> **📖 学習ガイドはこちら**: [{feature}_guide.md]({feature}_guide.md)（先にガイドで学んでから挑戦推奨）
```

## 2. ガイドの構成

### 2.1 必須セクション

1つのフェーズ（または1つの機能単位）には以下のセクションを全て含める：

```markdown
## {Phase名}: {タイトル}

> **ショートディスクリプション**

### 🎯 目標
- このフェーズで何ができるようになるか
- 具体的で計測可能な目標を箇条書き

### 📚 新しい概念
- 登場する用語や仕組みの解説
- 「なぜ」を中心に書く（WHAT より WHY）
- 図や表を使って視覚化する

### 🔧 必要な部品
- 必要な関数・型・設定項目の一覧
- 各関数のパラメータ表（必須／任意・型・説明）
- 戻り値の説明
- 呼び出し方の例（疑似コードや一部分のみ）

### 🔍 似ているもの同士の比較
- 混同しやすい機能・用語を表で比較
- 比較軸: 役割・引数・戻り値・使う場面

### 📝 自分で書いてみよう
- コードを書くためのヒント付き課題
- Step 形式で段階的に示す
- 答えは答案コード集に委ねる

### ✅ チェックポイント
- 正しく実装できたか確認する方法
- チェックリスト形式（[ ]）
```

### 2.2 全体に必須のセクション

ドキュメント全体として以下を含める：

| セクション | 場所 | 説明 |
|-----------|------|------|
| ロードマップ | 冒頭 | 全フェーズの一覧とゴール |
| このガイドの使い方 | 冒頭 | 学習の進め方を説明 |
| エラーが起きたときの調べ方 | 末尾付近 | よくあるエラーと対処 |
| おわりに | 末尾 | 学んだことのまとめ・全体を通しての気づき |
| 参考リンク | 末尾 | 公式ドキュメントや関連資料へのリンク |


### 2.3 見出しの直後には `<br>` を入れる

H1 / H2 / H3 の見出し行の直前、直後には、必ず `<br>` タグを1行挿入する：

```markdown
## Phase 1: データベース基盤
<br>
> **3つのテーブルを Drizzle ORM で定義し、マイグレーションします。**
```

**なぜ必要か**: 見出しの直後にテキストがくっつくと、行間が詰まって読みづらくなるため。
特に **H3 → 太字（`**...**`）→ 本文** のような連続は文字がギチギチに見える。<br>
タグ1行で視覚的な余白が生まれ、見出しと内容の区別が明確になる。

## 3. コードの扱い

### 3.1 ガイドに書いていいコード

以下の場合のみ、コードブロックを許容する：

**✅ OK: 疑似コード**
```typescript
// 疑似コード — 実際には動かない
export async function someFunction() {
  // 1. 入力を検証する
  // 2. DB からデータを取得する
  // 3. 処理結果を返す
}
```

**✅ OK: 関数シグネチャだけ**
```typescript
function createUploadUrl(options: {
  fileName: string;
  fileSize: number;
  contentType: string;
  expiresIn?: number;
}): Promise<{ uploadUrl: string; fileId: string }>
```

**✅ OK: 一部の抜粋（実装の核となる部分のみ、コメントで補足）**
```typescript
// 署名付きURLの生成（要注目ポイント）
const signedUrl = await r2.put(file.key, {
  httpMetadata: { contentType: file.mimeType },
  customMetadata: { uploadedBy: userId }, // ← メタデータにユーザー情報を埋め込む
});
```

**✅ OK: 型定義や定数の参照**
```typescript
import type { R2Bucket } from '@cloudflare/workers-types';
```

### 3.2 ガイドに書いてはいけないもの

**❌ NG: 完成コード（そのままコピペして動くもの）**
```typescript
// NG: これは答案コード集に書くべき
export async function uploadFile(file: File, userId: string) {
  const key = `${userId}/${crypto.randomUUID()}-${file.name}`;
  await r2.put(key, file.stream(), { httpMetadata: { contentType: file.type } });
  return { key };
}
```

**❌ NG: import の列挙（参照として必要な最小限のみ）**
→ 完成コードにしか出てこない多数の import は答案コード集に委ねる

### 3.3 答案コード集の構成

答案コード集は以下のルールで書く：

1. **フェーズ順 + ファイルパス** で整理する
2. 各コードブロックの直前に **ファイルパスと簡単な説明** を書く
3. コードは **コピペしてそのまま使える完成形** にする
4. 必要に応じてコメントで補足説明を入れる

````markdown
### `project/apps/api/src/repositories/fileRepository.ts`

```typescript
// 完成コード（そのまま動く）
export async function createFileRecord(data: { ... }) {
  ...
```
````

## 4. 解説の書き方

### 4.1 「なぜ」を書く

機能や設計の説明では、**「なぜそうするのか」** を必ず含める：

| 観点 | 悪い例 | 良い例 |
|------|--------|--------|
| ファイル保存先 | R2 に保存する | R2 に保存する。なぜなら D1 の容量制限（100MB）を超えるファイルを扱え、CDN 経由で高速配信できるから |
| アップロードURL | 直接 PUT させる | 署名付き URL を発行してから PUT させる。なぜならサーバーを経由せずクライアントから直接 R2 にアップロードでき、サーバー負荷と転送コストを削減できるから |
| Cookie 属性 | httpOnly を true にする | httpOnly を true にして XSS 対策をする。JavaScript から読み取れないようにすることで、仮にスクリプト注入があってもセッションが漏洩しない |

### 4.2 比較を入れる

「似ているけど違うもの」は積極的に比較表にする：

| 比較対象 | 比較軸の例 |
|---------|-----------|
| `createUploadUrl` vs `createDownloadUrl` | 引数・戻り値・使う場面・有効期限 |
| 署名付きURL vs 公開URL | セキュリティ・有効期間・認証要件 |
| アップロードフロー vs ダウンロードフロー | 段数・検証項目・R2操作 |
| ファイル一覧取得 vs ファイル個別取得 | レスポンス形式・ページネーション・キャッシュ戦略 |

### 4.3 段階的に抽象度を下げる

「全体像 → 詳細 → コード」の順で情報を提示する：

```
① 図や表で全体像を示す（このフェーズで何を作るか）
② 新しい概念を説明する（なぜそれが必要か）
③ 必要な関数や部品を列挙する（WHAT）
④ 各関数の使い方を解説する（HOW）
⑤ 自分で書くためのヒントを与える（課題）
```

### 4.4 学習者のつまずきポイントを先回りする

いわゆる「ハマりどころ」は積極的に書く：

```
**💡 罠**: `wrangler d1 migrations apply` は `project/apps/api/.env` を自動ロードしません。
`wrangler.toml` の `[vars]` に必要な変数を明示的に定義するか、`--env` フラグで環境を指定する必要があります。
```

## 5. 答案コード集の構成ルール

### 5.1 ファイル構成

答案コード集は以下の構成で書く：

````markdown
# {機能名} 答案コード集

> **このファイルの使い方**: [学習ガイド]({feature}_guide.md) で学習を進め、行き詰まったら該当するフェーズの答案を参照してください。

## 📋 フェーズ別コード一覧

| フェーズ | ファイル | 内容 |
|---------|---------|------|
| Phase 0 | `src/path/to/file.ts` | 説明 |
| ... | ... | ... |

## {Phase名}: {タイトル}

### `src/path/to/file.ts`

```typescript
// 完成コード
```

---
````

### 5.2 コードの掲載ルール

- **フェーズ番号順** に掲載する
- 1ファイルにつき1コードブロック（同じフェーズ内で複数ファイルあれば複数ブロック）
- コードブロックの直前に **ファイルパス** を `### ` 形式で書く
- コードはコピペして動く **完成形** にする
- 補足コメントはコード内のコメントとして書く

## 6. ガイドの配置とナビゲーション

### 6.1 配置先

ガイド系ドキュメントは以下のルールで配置する：

| カテゴリ | 配置先 | 例 |
|---------|--------|----|
| アーキテクチャ解説 | `doc/materials/architecture/` | `doc/materials/architecture/system/app-overview.md` |
| 環境構築手順 | `doc/materials/guide/` | `doc/materials/guide/environment-setup.md` |
| 実装ガイド | `doc/materials/implementation/` | `doc/materials/implementation/file-upload-guide.md` |
| UI/UX設計 | `doc/materials/design/` | `doc/materials/design/screen-requirements.md` |

### 6.2 プレビューサーバーとの連携

- `doc/serve.js` は `doc/materials/` 以下のファイルを自動的に配信する。
- ガイド作成後は `bash doc/serve.sh` でプレビューサーバーを起動し、ブラウザで表示を確認することを推奨する。
- 新しいガイドは `doc/src/` のプレビューアプリから参照できるよう、必要に応じて `doc/src/script.js` のナビゲーションに追記してよい。

### 6.3 従来のハンドブックがある場合

従来のハンドブック（`*_handbook.md`）が存在する場合は、ガイド・答案コード集・ハンドブックの冒頭で相互リンクする：

```markdown
<!-- ガイドの冒頭 -->
> **📖 従来のハンドブックはこちら**: [{feature}_handbook.md]({feature}_handbook.md)（完全版・リファレンス向け）

<!-- ハンドブックの冒頭 -->
> **📝 学習ガイドはこちら**: [{feature}_guide.md]({feature}_guide.md)（コードなしで学びたい方向け）
> **📝 答案コード集はこちら**: [{feature}_answers.md]({feature}_answers.md)（実装例を確認したい方向け）
```

## 7. フォーマットチェックリスト

新しくガイドを作成するときは以下を確認する：

### ガイド（`xxx_guide.md`）
- [ ] **見出し後の `<br>`**: H1/H2/H3 の直前、直後に `<br>` を入れて読みやすい行間を確保している
- [ ] ガイドと答案コード集のリンクが冒頭にある
- [ ] ロードマップ表がある
- [ ] 「このガイドの使い方」セクションがある
- [ ] 全フェーズが「目標・概念・部品・比較・課題・チェックポイント」で構成されている
- [ ] 完成コードが含まれていない（疑似コードや一部抜粋のみ）
- [ ] 「なぜ」の説明が含まれている
- [ ] 似ている機能の比較表がある
- [ ] ハマりどころが記載されている
- [ ] エラー時の調べ方が記載されている

### 答案コード集（`xxx_answers.md`）
- [ ] ガイドへのリンクが冒頭にある
- [ ] フェーズ別コード一覧表がある
- [ ] 全コードがファイルパス付きで掲載されている
- [ ] コードはコピペして動く完成形である

### 従来のハンドブック（`xxx_handbook.md`）
- [ ] 冒頭にガイドと答案コード集への案内がある（存在する場合）

### 配置・ナビゲーション
- [ ] 適切な `doc/materials/` 配下のディレクトリに配置されている
- [ ] プレビューサーバーで正しく表示されることを確認した
